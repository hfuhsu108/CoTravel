## 批次 A：錯誤回饋與誤刪防護

### A1. 地圖分頁寫入失敗零回饋（原 #3，最重要）

**檔案**：`src/routes/tabs/MapTab.tsx`
**現況**：`error` state 有 10 多處寫入（更新 602、刪除 620、複製 650、移動 665、候選勾選 676、移除候選 687、移除交通 722、拖曳排序 801、定位 440/455、POI 873），但唯一渲染處是 `if (error && !trip)`（820 行）的全螢幕錯誤——旅程載入後所有操作失敗都無聲無息。
**步驟**：
1. 保留 `error && !trip` 的全螢幕錯誤（初載失敗仍用它）。
2. 加自動清除：`useEffect`——當 `error && trip` 時起 4 秒 timer 呼叫 `setError(null)`，cleanup 時清 timer。
3. 在地圖容器內（主 return 裡）渲染 toast：`{error && trip && (<div ...>{error}</div>)}`。定位與樣式**比照 `src/components/ActivityBanner.tsx`**（絕對定位、頂部置中、`calc(env(safe-area-inset-top) + 60px)` 避開 MapTopBar、z-[60]），但配色改警示紅系（`bg-danger` 系 token，參考全螢幕錯誤的用色）。
4. 不要新增全域元件、不要動各 `setError` 呼叫點——它們已經寫對了，只是沒人渲染。

**驗收**：build+preview → DevTools Network 設 Offline → 在地圖分頁嘗試刪除一個行程項目 → 頂部出現紅色錯誤提示、4 秒後自動消失；恢復 Online 後操作正常。

### A5. push-handler 無 try/catch（原 #9）

**檔案**：`public/push-handler.js:3-15`
**現況**：`event.data.json()` 遇非 JSON payload 會 throw，整個 push 事件失敗、通知不顯示（違反 userVisibleOnly 承諾，瀏覽器可能降級訂閱）。
**步驟**：解析包 try/catch，失敗時 fallback：

```js
let payload
try {
  payload = event.data ? event.data.json() : {}
} catch {
  payload = {}
}
const title = payload.title || '同行提醒'
```

其餘顯示邏輯照舊（body、icon 等欄位同樣給預設值）。**注意**：此檔是 service worker 腳本，改完要 build 才會進 SW；語法保持原生 JS（無 import）。

**驗收**：`cmd /c npm run build` 通過；線上部署後推播照常顯示（完整驗證需等下次提醒觸發，先以 build 通過＋code review 為準）。

## 批次 C：基礎設施（本批次需要使用者配合手動步驟，開工前先讓使用者知道）

### C1. Edge Function 無認證＋重複推播競態（原 #7）

**檔案**：`supabase/functions/check-reminders/index.ts`（入口檢查 45-49、主迴圈 53-141）
**步驟**：
1. **認證**：函式開頭加——`req.headers.get('x-cron-secret') !== Deno.env.get('CRON_SECRET')` 時回 401。
2. **先佔後發（claim）**：把「查詢未 fired → 逐一發送 → 最後批次標 fired」改為「先原子標記再發送」：

```ts
// 先把到期且未發的提醒原子標為 fired 並取回（兩個併發執行只會各拿到不重疊的列）
const { data: claimed, error } = await supabase
  .from('reminders')
  .update({ fired: true })
  .eq('fired', false)
  .lte(/* 既有的到期條件，照原查詢搬過來 */)
  .select()
```

3. 對 claimed 逐一發送；**單筆發送失敗**時把該筆 `fired` 改回 false（下次重試），並 console.error——寧可下分鐘重發，不可無聲遺漏。
4. **使用者手動步驟**（執行模型負責產出精確指令與 SQL，交使用者執行）：
   - 產一組隨機密鑰，在 Supabase 設定：`supabase secrets set CRON_SECRET=<密鑰>`（或 Dashboard → Edge Functions → Secrets）。
   - 找出 pg_cron 排程定義（先查 `supabase/schema.sql` 第 9 節附近有沒有；沒有就寫一段查詢＋更新 cron job 的 SQL 給使用者在 SQL Editor 執行），讓 `net.http_post` 的 headers 加上 `x-cron-secret`。
   - 重新部署函式：`supabase functions deploy check-reminders`。
5. 改動 `supabase/functions/` 下的檔案即可，**不要動 schema.sql 的表定義**。

**驗收**：部署後——不帶 header 的 POST 回 401；帶正確 header 手動觸發一次成功；等一次真實提醒確認只收到一則推播。

### C2. 離線快取無版本／失效機制（原 #8）

**檔案**：`src/lib/offline/itineraryCache.ts`、`src/lib/offline/docCache.ts`；若離線工作單已完成，加上 `docsMetaCache.ts`
**步驟**：
1. 快照檔各加一個**手動維護**的常數 `const SNAPSHOT_SCHEMA_VERSION = 1`，快照介面加 `schemaVersion: number` 欄位；`saveSnapshot` 寫入、`getSnapshot` 讀到 `schemaVersion !== SNAPSHOT_SCHEMA_VERSION`（含 undefined，即舊資料）時回 null（視同無快照）。日後快照結構變動時，改結構的人負責 bump 這個常數。
   - **為什麼不用 APP 版本號**：APP 版本每次部署都變，會把還沒重新上線同步的使用者的離線快照全數作廢，反而傷害離線體驗。
   - **不要動 IndexedDB 的 `VERSION`（DB 版本）**——結構沒變，不需要 onupgradeneeded 遷移。
2. `docCache`（文件 Blob）孤兒清理：用 Grep 找 `removeDocument`（`src/lib/documents.ts`）的所有呼叫端，確認刪文件時有呼叫 `removeCached(storage_path)`；漏了就在呼叫端補上（照該檔既有註解「由呼叫端移除」的設計，不要改進 lib 層）。
3. 下載文件寫入快取處（呼叫 `cacheDoc` 的地方）：catch `QuotaExceededError` 給明確訊息「裝置儲存空間不足，無法離線保存」。

**驗收**：手動在 DevTools → Application → IndexedDB 把某快照的 `schemaVersion` 改掉（或用舊資料）→ 離線讀取視同無快照、線上重載後恢復；刪除一份已下載文件 → IndexedDB 內對應 Blob 同步消失。
