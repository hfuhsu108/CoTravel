# 02 · 技術架構與費用

## 架構總覽

```
┌─────────────────────────────────────────────┐
│  前端 PWA（React + Vite + TS, Tailwind）       │
│  ├─ Google Maps JS API（地圖/搜尋/Directions）  │
│  ├─ Supabase JS SDK（資料/認證/即時/儲存）      │
│  └─ Service Worker（PWA 離線快取：文件+行程）    │
└───────────────┬─────────────────────────────┘
                │
        ┌───────┴────────┐
        │   Supabase      │
        │  ├─ Auth        │  帳號登入、兩人配對
        │  ├─ Postgres    │  行程/景點/交通/文件中繼/行李
        │  ├─ Realtime    │  改動通知
        │  └─ Storage     │  文件檔案（透過 storage 抽象層）
        └─────────────────┘

        ┌─────────────────┐
        │ Google Maps      │  地圖底圖、地點搜尋、路線規劃
        │ Platform         │
        └─────────────────┘

  託管：GitHub Pages（純靜態 PWA，網址含 /<repo>/ 子路徑）
```

## 技術選型

| 項目 | 選用 | 理由 |
|---|---|---|
| 框架 | React + Vite + TypeScript | 生態成熟、PWA 工具齊全、Claude Code 熟悉 |
| PWA | vite-plugin-pwa（Workbox） | 離線快取文件與行程 |
| 樣式 | Tailwind CSS | 快速、與 prototype 對齊 |
| 地圖元件 | `@vis.gl/react-google-maps` | Google 官方推薦的 React 封裝 |
| 後端 | Supabase | 一站式 Auth+DB+Realtime+Storage，免費起步、不綁卡 |
| 拖拉排序 | `@dnd-kit/core` | 行程項目跨天拖拉 |
| 託管 | GitHub Pages | 純靜態、免費、使用者已熟悉；本架構無自有伺服器，靜態託管即可 |

## 費用（兩人自用）

| 服務 | 用途 | 實際費用 | 綁卡 |
|---|---|---|---|
| Supabase 免費方案 | 登入、DB、即時、檔案 | 免費（500MB DB + 1GB 檔案） | 否 |
| GitHub Pages | 託管 | 免費 | 否 |
| Google Maps Platform | 地圖 + 搜尋 + 路線 | 用量遠低於免費額度 ≈ 0 | **是**（須設預算上限） |

> **唯一綁卡風險**在 Google Maps。開通時務必在 Google Cloud Console 設「預算上限 + 用量警報」，並對 API 金鑰加 HTTP referrer 限制，避免外洩被盜刷。詳見 `docs/06`。

## 關鍵決策

### 1. 檔案儲存「可抽換」（重要）

- 起步用 **Supabase Storage（1GB 免費）**。
- 文件多是小檔（機票 QR、PDF、簽證圖，約 1–3MB），1GB ≈ 10–30 趟，頭幾年夠用。
- **所有檔案存取必須透過 `src/lib/storage/` 抽象介面**（如 `uploadFile`、`getFileUrl`、`deleteFile`），元件不可直接呼叫 Supabase Storage。
- 接近滿時，只需替換抽象層的實作，**無痛升級 Cloudflare R2（免費 10GB、天生共用桶）**，不影響行程資料。

### 2. Supabase 閒置自動暫停

- 免費專案 7 天無活動會暫停（資料不掉，手動喚醒）。兩趟旅行間可能被暫停 → 接受手動喚醒；真的嫌煩再考慮升級 Pro（$25/月，不暫停）。

### 3. 為何不用 Google Drive 當儲存

- Drive 15GB 雖大，但小檔用途用不到；且要扛 OAuth + 共享資料夾複雜度、行程資料仍得放 Supabase。CP 值不如 R2。

### 4. 即時同步用「通知」而非「逐字協作」

- 兩人自用、且使用者只要「改動通知」→ 用 Supabase Realtime 訂閱資料表變更 + last-write-wins，**不需** CRDT/OT，大幅降低複雜度。

### 5. 託管用 GitHub Pages（注意子路徑）

- 使用者帳號 `hfuhsu108`，網址為 `https://hfuhsu108.github.io/<repo>/`。
- 因網址帶 `/<repo>/` 子路徑，**Vite 須設 `base: '/<repo>/'`**，且前端路由要用對應 basename（建議 HashRouter 或設定 base 的 BrowserRouter），否則重新整理會 404。
- PWA 的 `scope` 與 manifest 的 `start_url` 也要對齊子路徑。
- 純靜態託管即可，因為後端全在 Supabase、地圖在 Google，無自有伺服器。
